## How I Fixed It / What I Did

Created a Python script `Paper/scripts/split_large_markdown_cases.py` that:
1. Scans all markdown files in `markdown_cases/`
2. Identifies files over 480,000 words (using 480K as a safe threshold below the 500K limit)
3. Splits each over-limit file into multiple parts at natural paragraph breaks (blank lines)
4. Saves parts as `{case_name}_part1.md`, `{case_name}_part2.md`, etc.
5. Each part starts with a header identifying it as a continuation
6. Outputs a summary of what was split and where

The split output is saved to `Paper/research/markdown_cases/split/` to keep things organized.

## Challenges Encountered

1. **Scale of the problem**: Files are not just slightly over — some are 2× the limit (Fisher at 1.16M words). This is because the markdown files contain the OCR of the *entire* court record, not just the Supreme Court opinion.
2. **OCR noise**: The content shows OCR artifacts (garbled characters, double-spaced words) which means simple word extraction isn't clean.
3. **Content structure**: These files include full trial transcripts, briefs from both sides, exhibits, appendices, etc. — essentially everything filed with the court.
4. **NotebookLM markdown support**: NotebookLM doesn't natively parse `.md` formatting — it treats it as plain text, so the markdown structure provides marginal benefit.
